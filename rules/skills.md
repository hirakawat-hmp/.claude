# Skill Activation Rules

スキルは description のみが常時ロードされ、本文（手順・テンプレート）は必要時にのみ読み込まれる。
そのため「いつ発動すべきか」の判断はエージェントに委ねられている。
以下は、ユーザーが明示的にスラッシュコマンドを打たなくても、メインエージェントが**自律的にスキルを発動すべき場面**を定めた規則である。

これらは `@rules/agents.md` の「Orchestration-Execution Pipeline」と一体で機能する。
各フェーズの入口・出口でスキルを発動し、規律ある開発を担保すること。

## 発動マトリクス（CRITICAL）

| 場面 / トリガー | 発動するスキル | フェーズとの対応 |
|---|---|---|
| 指示が曖昧・包括的（「いい感じに」「自由に」「最適と思う方針で」「最強の〇〇に」）、または新規機能・新規PJ・大規模実装に着手する直前 | `requirements-facilitator` | 1. 調査/計画の**前** |
| アーキテクチャ・技術スタック・ライブラリ・データモデルなど、トレードオフを伴う重要な技術判断を下した直後 | `decision-logger` | 2. 実装フェーズの判断時 |
| 合意済みの仕様・方針・データモデル・公開APIを変えようとするとき、または途中で要件が追加・変更されたとき（「やっぱりこうしたい」「この概念を消したい」「これもMUSTに」） | `change-impact-gate` | 全フェーズ横断（着手の**前**） |
| セッション終盤、作業の大きな区切り、中断するとき、または進捗をまとめる必要があるとき | `session-state` | 4. レビュー後〜セッション終了時 |

## 運用原則

1. **着手前ゲートを最優先する** — `requirements-facilitator` と `change-impact-gate` は「手を動かす前」に発動する。曖昧なまま、または影響範囲を確認しないまま実装委譲してはならない。
2. **重要判断は必ず残す** — トレードオフを伴う選択をしたら、ユーザーに報告する前に `decision-logger` で `docs/decisions/` に記録する。
3. **過剰発動を避ける** — 要件が明確な小タスク、タイポ修正、些細なスタイル調整には発動しない。スキルの description にある「使わない場面」を尊重する。
4. **スキルとサブエージェントを連携させる** — 例: `change-impact-gate` の影響範囲調査は `@codebase-researcher` に委譲し、その結果をもとにゲートを提示する。スライド生成は `requirements-facilitator` で仕様を握ってから `@slide-builder` に委譲する（`@rules/agents.md` の成果物生成フェーズを参照）。
5. **STATE.md は各プロジェクトに残す** — `session-state` は作業中プロジェクトの `.claude/STATE.md` に書く。グローバル `~/.claude/` に書かない。SessionStart hook が次回これを自動でコンテキスト注入する。

## なぜこの規則が必要か

これらのスキルは、過去のプロンプト分析で繰り返し観測された課題（曖昧委任→迷走、途中の方針変更→手戻り、セッション間の状態喪失）への対策である。
スキルの存在をルールとして常駐させることで、ユーザーが毎回スラッシュコマンドを打たなくても、メインエージェントが自律的に正しい場面でスキルを発動できるようにする。
