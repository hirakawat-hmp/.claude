# change-impact-gate

## 目的

途中での方針変更・要件後出しによる手戻りと破壊を防ぐ。変更に飛びつく前に影響範囲を明示し、ユーザーの承認を得るゲートとして機能する。

## 使うべき場面・使わない場面

**使う**: 以下のいずれかに該当するとき

- 合意済みの仕様・データモデル・公開 API・アーキテクチャ方針を変更しようとしている
- 作業中に「やっぱりこうしたい」「これも追加で」「あの決定を覆したい」という変更依頼を受けた
- MUST 要件が追加・削除・変更された
- 変更が複数ファイル・複数モジュールにまたがる可能性がある

**使わない**: 以下のケースは過剰なゲートになるため使わない

- タイポ・コメントの修正など、動作に影響しない変更
- まだ実装が始まっていない（合意前の）計画段階での調整
- 影響が1ファイル以内に完全に閉じることが明らかな小変更

---

## ゲート手順

### ステップ1: 変更の把握

変更依頼を受けたら、まず以下を整理する。

- **何を変えるか**: 変更対象の機能・概念・インターフェース名
- **何から何へ**: 変更前の状態 → 変更後の状態

### ステップ2: 影響範囲の調査

影響するファイル・モジュール・エージェントを特定する。コードベースが大きい場合や影響が不明確な場合は **codebase-researcher** に調査を委譲する。

調査観点:

- 変更対象を参照・依存しているファイル
- 変更によって壊れる可能性のある既存の動作・テスト
- 連携する外部システム・API・データストアへの影響
- ドキュメント・ADR など、記録済みの決定事項との整合

### ステップ3: 影響サマリーの提示

以下のフォーマットでユーザーに提示し、承認を得てから着手する。

```
## 変更影響レビュー

### 変更の要約
- 対象: <機能・概念・インターフェース名>
- 変更内容: <変更前> → <変更後>

### 影響を受けるファイル/モジュール
- <ファイルパス or モジュール名> — <影響の内容>
- ...

### 破壊的変更
- [ ] 既存の動作への影響: <あり / なし / 要確認>
- [ ] 既存データへの影響: <あり / なし / 対象外>
- [ ] 他機能・他モジュールへの波及: <あり / なし / 要確認>

### 必要な作業量
- 小（数行〜1ファイル） / 中（複数ファイル） / 大（アーキテクチャ全体）

### 代替案
- （変更せずに目的を達成できる別の方法があれば提示）

### 巻き戻しの容易さ
- 容易 / 困難 / 不可（理由: ）

---
この変更を進めてよいですか?
```

### ステップ4: 承認後の後処理

ユーザーが承認した場合:

1. 影響を受ける `docs/requirements/` のファイルを更新する
2. 既存の ADR が無効になる場合は **decision-logger** を使って新しい ADR を作成し、旧 ADR のステータスを「置換: NNNN」に更新する
3. 実装を **implementer** に委譲する

ユーザーが却下した場合:

- 元の合意内容を維持して作業を続ける
- 却下の理由をメモとして残す（任意）

---

## 他スキルとの連携

- 変更に伴い要件を再定義する必要がある場合は **requirements-facilitator** を再起動する
- 変更の結果として新しい技術判断が生まれた場合は **decision-logger** で ADR を作成する
