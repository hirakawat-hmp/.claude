# decision-logger

## 目的

「なぜこの設計にしたか」を後から追えるようにする。セッションをまたいでも判断の経緯が失われない。

## 記録する/しない の判断基準

**記録する**（以下のいずれかに該当）:

- 複数の代替案を検討してひとつを選んだ
- トレードオフが存在し、何かを犠牲にする選択をした
- 将来の自分や他のエージェントが「なぜ?」と疑問に思いそうな選択
- アーキテクチャ・技術スタック・データモデル・公開API設計に関わる判断
- 一度決めると覆しにくい（高コストな）選択

**記録しない**（自明な実装詳細）:

- 慣用的な書き方の選択（変数名、インデントなど）
- 既存の方針を単純に踏襲した実装
- 試行錯誤の途中段階（確定した判断のみ記録）

## ADR 作成手順

### ステップ1: 連番の採番

`docs/decisions/` ディレクトリ内の既存ファイルを確認し、最大の連番 N を特定する。新しい ADR は N+1 番とする。ディレクトリが存在しない場合は `mkdir -p docs/decisions` を実行し、0001 から開始する。

```bash
ls docs/decisions/ 2>/dev/null | grep -oE '^[0-9]+' | sort -n | tail -1
```

### ステップ2: ファイル作成

ファイル名: `docs/decisions/NNNN-<slug>.md`

- NNNN は4桁ゼロ埋め（例: 0001, 0012, 0123）
- slug はタイトルを英小文字・ハイフン区切りに変換（例: `use-postgresql-for-main-db`）

### ステップ3: 以下のテンプレートで記述する

```markdown
# NNNN. <タイトル>

## ステータス

採用

## 日付

YYYY-MM-DD

## 文脈

（なぜこの決定が必要だったか。背景・制約・解決したい問題を記述）

## 決定

（何を決めたか。簡潔に1〜3文で）

## 検討した代替案

- **案A: <名称>** — <概要と、採用しなかった理由>
- **案B: <名称>** — <概要と、採用しなかった理由>

## 結果・トレードオフ

（この決定によって得られるもの・失われるもの・注意すべき副作用）

## 関連

- 関連する ADR や要件ドキュメントへのリンク（あれば）
```

### ステータスの値

| 値 | 意味 |
|---|---|
| 提案中 | まだ合意が取れていない |
| 採用 | 合意済みで実施中 |
| 却下 | 検討したが採用しなかった |
| 置換 | 後の ADR によって置き換えられた（`置換: NNNN` と記載） |

---

## 記述上の注意

- 1つの ADR に1つの決定のみ記録する。複数の判断をまとめない
- 「文脈」には決定当時の状況を書く。後から読む人が背景を理解できるよう具体的に
- 未来のことを予測して書かない。決定した時点での事実と判断のみ
- 簡潔に。1 ADR は50行以内を目安とする

---

## 他スキルとの連携

- 重要な技術判断に至る前に要件が曖昧な場合は **requirements-facilitator** で要件を固めてから記録する
- 既に記録済みの ADR の内容を変更する場合は **change-impact-gate** を起動して影響範囲を確認する
