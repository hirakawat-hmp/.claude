# requirements-facilitator

## 目的

曖昧な委任のまま実装に突入してエージェントが迷走・手戻りするのを防ぐ。着手前に最小限の要件を合意する。

## 使うべき場面・使わない場面

**使う**: 以下のいずれかに該当するとき

- 「いい感じに」「自由に」「最適な方法で」「最強の」など、完了条件が不明確な委任を受けた
- 新規機能・新規プロジェクト・大規模実装（影響ファイル5件以上が見込まれる変更）の着手前
- ゴール状態・制約・品質基準のいずれかが曖昧なまま実装計画を立てようとしている

**使わない**: 以下のケースには過剰な確認になるため使わない

- タスクが小さく、要件が一文で明確に表現されている（例:「このファイルのtypoを直して」）
- 以前のセッションで要件が `docs/requirements/` に記録済みで、それを踏襲する作業

---

## ファシリテーション手順

### ステップ1: 不明項目の特定

以下のチェックリストを確認し、不明または曖昧な項目を洗い出す。

1. **ゴール状態**（何が「完成」か。完了の定義は何か）
2. **制約**（技術スタック・やってはいけないこと・既存資産との整合）
3. **優先度**（MUST/SHOULD/COULD の区別。何を犠牲にしてよいか）
4. **品質基準**（パフォーマンス・見栄え・テストカバレッジ等の具体的な数値目標）
5. **非ゴール**（今回やらないこと・スコープ外の明示）
6. **想定ユーザー・利用文脈**（誰が、いつ、どういう状況で使うか）

### ステップ2: 対話で引き出す

- 一度に全項目を聞かない。不明な項目を2〜4問ずつまとめて質問する
- ユーザーが既に答えている項目は再質問しない
- 選択肢を提示すると答えやすい（例:「A案とB案どちらが近いですか?」）
- 「わからない / 任せる」という回答も有効な情報として受け取り、その場合はエージェント側がデフォルト値を提案して確認を取る

### ステップ3: 要件サマリーの提示と合意

全項目が埋まったら、以下の形式で要件サマリーを提示しユーザーの承認を得る。

```
## 要件サマリー: <トピック名>

### ゴール状態
- （完了の定義）

### 制約
- （技術・設計・方針上の制約）

### 優先度
- MUST: ...
- SHOULD: ...
- COULD: ...

### 品質基準
- （数値目標や基準）

### 非ゴール（今回やらないこと）
- ...

### 想定ユーザー・利用文脈
- ...
```

「この内容で進めてよいですか?」と確認し、承認を得てから実装フェーズに移る。

### ステップ4: 成果物の保存

合意した要件を以下のパスに保存する。

```
docs/requirements/<topic-slug>.md
```

> **注意**: 必ず作業中プロジェクトのルートからの相対パスで保存すること。`~/.claude/docs/requirements/` へのグローバル保存は誤りであり、絶対に行ってはならない。

ファイルが存在しない場合は新規作成。既存ファイルがある場合は追記または上書きし、最終更新日を記録する。

**ファイル冒頭フォーマット例:**

```markdown
# 要件: <トピック名>

最終更新: YYYY-MM-DD

## ゴール状態
...

## 制約
...

## 優先度
- MUST:
- SHOULD:
- COULD:

## 品質基準
...

## 非ゴール
...

## 想定ユーザー・利用文脈
...
```

## 他スキルとの連携

- 要件合意後に重要な技術選定が発生した場合は **decision-logger** を使って判断を記録する
- 合意済み要件を後から変更しようとする場合は **change-impact-gate** を起動して影響範囲を確認する
